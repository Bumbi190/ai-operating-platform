## Canonical Text

### 20.1 Purpose of This Chapter

Executive Intelligence must convert decisions into executable organizational work.
A recommendation is not a mission.
A roadmap milestone is not a mission.
A priority is not a mission.
A founder instruction is not necessarily a sufficiently defined mission.
Before Manager and Workforce can act, Omnira must translate executive intent into a bounded execution contract.
That contract is the Executive Mission Brief.
A Mission Brief should make it possible to answer:
What must be achieved?
Why does it matter?
Which project owns the work?
What does success look like?
What is inside scope?
What is outside scope?
Which authority exists?
Which tools and data may be used?
Which actions are forbidden?
Where are the approval gates?
What is the deadline?
How should progress be reported?
When should work stop or escalate?
The Mission Brief is the bridge between strategy and execution.

### 20.2 The Core Thesis

Executive Intelligence should define the outcome.
Manager should coordinate the execution.
Workforce should perform the work.
The canonical separation is:
Executive Intelligence:
Defines purpose, outcome, constraints, authority, and success.
Manager:
Plans, routes, sequences, and supervises execution.
Workforce:
Executes bounded work and reports evidence.
Governance:
Determines what may be done.
Founder:
Retains final authority where required.
The foundational principle is:
Workforce should never need to invent the strategic meaning, authority, or success criteria of its own mission.

### 20.3 Mission Brief as an Execution Contract

A Mission Brief is more than a prompt.
It is a structured contract.
It should define:
Objective.
Context.
Desired outcome.
Success criteria.
Scope.
Constraints.
Authority.
Resources.
Tools.
Forbidden actions.
Approval gates.
Deadline.
Reporting.
Escalation.
Completion conditions.
The Mission Brief should remain understandable to humans and executable by systems.

### 20.4 Mission Brief vs Prompt

A prompt instructs a model.
A Mission Brief governs organizational work.
A prompt may say:
Analyze our publishing workflow.
A Mission Brief should say:
Project:
The Prompt.
Objective:
Determine why short-news publication fails to persist reliably.
Success:
Root cause identified, evidence provided, safe fix plan prepared.
Constraints:
Read-only investigation. No production changes.
Approval Gate:
Implementation requires founder approval.
Deadline:
Before the next publishing window.

### 20.5 Mission Brief vs Task

A Mission Brief defines a meaningful outcome.
A task is a specific unit of work inside the mission.
Example:
Mission:
Validate project isolation before broader autonomy.
Tasks:
Audit RLS.
Test cross-project queries.
Inspect credentials.
Review cron behavior.
Produce remediation plan.
Manager may create tasks from the mission.

### 20.6 Mission Brief vs Roadmap Milestone

A roadmap milestone describes what must become true.
A Mission Brief delegates work intended to make it true.
Example:
Milestone:
Approval architecture is operational and validated.
Mission:
Design and validate a project-scoped Approval Inbox with decision-ready context, action controls, and auditability.

### 20.7 Mission Brief vs Decision

A decision authorizes direction.
A Mission Brief operationalizes that direction.
Example:
Decision:
The Prompt will become the first autonomy proving ground.
Mission:
Prepare the short-news workflow for a bounded L4 publishing trial.

### 20.8 Mission Brief vs Workflow

A workflow is a repeatable execution mechanism.
A Mission Brief may:
Create a workflow.
Use an existing workflow.
Modify a workflow.
Validate a workflow.
Retire a workflow.
The mission remains the higher-level executive contract.

### 20.9 Mission Brief vs Project

A project is an enduring organizational scope.
A mission is temporary and outcome-specific.
A project may contain many missions.

### 20.10 Mission Brief vs Role

A Workforce role describes a reusable capability.
A mission assigns that capability to a bounded objective.
Example:
Role:
Editorial Researcher.
Mission Assignment:
Validate sources for The Prompt short-news autonomy trial.

### 20.12 Strategic Mission

Prepare a portfolio decision.
Compare strategic alternatives.
Design a roadmap.

A Build Mission creates or changes a capability.

Build an Approval Inbox.
Create a performance collector.
Implement project-scoped memory retrieval.
Prepare a new publishing workflow.

### 20.14 Investigation Mission

Evidence requirements.
Systems to inspect.
Questions to answer.

### 20.15 Validation Mission

A Validation Mission determines whether an existing capability is safe, correct, or ready.

Validate rollback.
Validate migration safety.
Validate publishing persistence.

### 20.17 Stabilization Mission

A Stabilization Mission improves:

Security.

Cost control.

### 20.18 Risk-Reduction Mission

A Risk-Reduction Mission addresses a known exposure.

Affected systems.

### 20.19 Recovery Mission

A Recovery Mission follows:

Failed rollout.

### 20.20 Learning Mission

Evidence.

### 20.21 Operational Mission

Run the monthly performance review.
Maintain content freshness.
Process standard support requests.

### 20.29 Executive Owner

The Executive owner is responsible for the mission’s strategic validity.

Portfolio Executive.
Project Executive.
Founder.
Authorized governance role.

### 20.30 Mission Owner

The Mission owner is accountable for delivering the mission outcome.
This may be:
Manager.
Workforce lead.
Human project owner.
Specialized agent team.
One mission should have one accountable owner.

### 20.31 Mission Objective

The objective should state what the mission is intended to achieve.
It should be:
Outcome-oriented.
Specific.
Strategically relevant.
Bounded.
Verifiable.
Weak:
Improve Memory.
Strong:
Resolve the Stage 1 Memory schema and ownership conflicts so implementation can proceed safely on the dedicated branch.

### 20.32 Strategic Context

Strategic context explains why the mission matters.
It may include:
Related objective.
Roadmap milestone.
Decision.
Risk.
Revenue opportunity.
Project mode.
Portfolio tradeoff.
Workforce should understand enough context to make correct local choices.

### 20.33 Context Minimization

The mission should include the context required for execution.
It should not expose unnecessary:
Founder-private information.
Other project data.
Sensitive strategy.
Unrelated Decision Ledger history.
Confidential customer details.

### 20.34 Expected Outcome

The expected outcome describes the future state that should exist after successful completion.
Example:
The publishing persistence defect is understood, a safe remediation is prepared, and no production changes have been made without approval.

### 20.35 Deliverables

Implementation plan.
Code change.
Test results.
Policy proposal.
Workflow configuration.
Decision package.

### 20.36 Success Criteria

Success criteria define how completion is judged.
They should be:
Observable.
Relevant.
Sufficient.
Non-contradictory.
Connected to the objective.

### 20.37 Quantitative Success Criteria

Examples:
All project-isolation tests pass.
Zero cross-project query leakage.
Correction rate below 5%.
Workflow completes thirty successful trial runs.
Build and typecheck pass.

### 20.38 Qualitative Success Criteria

Examples:
The root cause is supported by evidence.
The recommendation is implementation-ready.
The policy is understandable and enforceable.
The user experience remains coherent.
Qualitative criteria should still be reviewable.

### 20.39 Minimum Success

A mission may define the minimum acceptable outcome.
Example:
Minimum:
Identify the root cause and produce a safe fix plan.
Target:
Implement and validate the fix after approval.

### 20.40 Target Success

Target success describes the desired complete result.

### 20.41 Stretch Success

Stretch outcomes may be included where they do not compromise the core mission.

### 20.243 Failure Modes

Executive Mission Briefs must prevent several failure modes.
Failure Mode 1 — Prompt as Mission
A vague model prompt replaces an execution contract.
Failure Mode 2 — Task List Without Outcome
Work is completed without achieving strategic value.
Failure Mode 3 — Undefined Authority
Workforce does not know what it may change.
Failure Mode 4 — Scope Creep
The mission expands without review.
Failure Mode 5 — Hidden Approval Gate
Work reaches a consequential action before approval requirements are known.
Failure Mode 6 — Manager Redefines Strategy
Operational coordination changes the objective.
Failure Mode 7 — Executive Micromanagement
The brief dictates every implementation detail.
Failure Mode 8 — Missing Success Criteria
Completion cannot be judged.
Failure Mode 9 — Wrong-Project Execution
Work uses the wrong data, credentials, or systems.
Failure Mode 10 — Missing Stop Conditions
Unsafe work continues.
Failure Mode 11 — Reporting Overload
Founder receives excessive operational detail.
Failure Mode 12 — Silent Mission Mutation
Scope or authority changes without versioning.
Failure Mode 13 — Zombie Mission
Inactive work remains Active indefinitely.
Failure Mode 14 — Completion Theatre
Tasks are marked done while the objective remains unmet.
Failure Mode 15 — Context Leakage
Unnecessary private or cross-project information reaches Workforce.
Failure Mode 16 — Orphaned Mission
No one remains accountable for the outcome.
These failure modes should be treated as organizational and architecture constraints.
