## Canonical Text

### 22.1 Purpose of This Chapter

Executive Intelligence cannot lead effectively if every conversation, decision, project, and mission begins from zero.
It must remember:
What the founder is trying to achieve.
Why earlier decisions were made.
Which strategies have already been tested.
Which project boundaries apply.
What Workforce has learned.
Which assumptions repeatedly fail.
Which preferences are stable.
Which risks recur.
What must not be forgotten.
However, memory also creates danger.
A remembered statement may be outdated.
A preference may have changed.
A project-specific lesson may be applied to the wrong project.
An inferred pattern may be mistaken for an approved rule.
A previous decision may have expired.
Private information may be exposed to Workforce.
Omnira therefore requires a formal integration between Executive Intelligence and Atlas Memory.
This chapter defines how Executive Intelligence should retrieve, interpret, use, validate, update, and restrict memory.
The purpose is continuity without hidden authority.

### 22.2 The Core Thesis

Memory should inform Executive judgment.
Memory should not replace:
Current evidence.
Active policy.
Decision authority.
The Decision Ledger.
Project scope.
Founder confirmation.
Governance.
The foundational principle is:
Memory provides context.
Memory does not grant permission.

### 22.3 Why Executive Intelligence Needs Memory

Without memory, Executive Intelligence would repeatedly:
Ask the same questions.
Reopen settled discussions.
Forget founder preferences.
Repeat failed experiments.
Misunderstand project history.
Lose roadmap continuity.
Reconstruct decisions from scattered chats.
Waste founder attention.
Memory allows Omnira to behave like a continuous organization rather than a sequence of isolated conversations.

### 22.4 The Executive Continuity Problem

Executive work extends across:
Days.
Weeks.
Months.
Years.
Projects.
Missions.
Reviews.
Incidents.
Strategy changes.
The system must preserve enough history to understand the present without replaying everything that has ever happened.

### 22.5 Memory Is Not a Transcript

A transcript records what was said.
Memory should preserve what remains useful.
Useful memory may include:
Stable facts.
Founder preferences.
Lessons.
Patterns.
Important context.
Project history.
Repeated constraints.
Known risks.
Successful methods.
Not every message should become durable memory.

### 22.6 Memory Is Not the Decision Ledger

The Decision Ledger preserves formal, authorized decisions.
Memory preserves useful context and learning.
Example:
Decision Ledger:
The Prompt is approved as Omnira’s first autonomy proving ground.
Memory:
The founder prefers small, reversible autonomy trials before broader expansion.
The distinction must remain explicit.

### 22.7 Memory Is Not Policy

A remembered instruction may influence recommendations.
It does not become active policy unless it has passed the required governance process.
Example:
Memory:
The founder usually prefers newsletters to be reviewed before sending.
Policy:
Newsletter sending requires human approval.
Only the policy governs action.

### 22.8 Memory Is Not Authority

A memory such as:
The founder previously approved this type of action.
does not authorize the next action.
Authority must come from:
Current approval.
Active policy.
Autonomy license.
Budget mandate.
Decision Ledger decision.
Valid delegated authority.

### 22.9 Memory Is Not Current Truth

Memory may become stale.
Examples:
A project mode changed.
A price changed.
A workflow was replaced.
A policy expired.
The founder changed preference.
A dependency completed.
A company strategy changed.
Memory must therefore be evaluated rather than blindly trusted.

### 22.10 The Memory Integration Objective

Executive Memory Integration should help Omnira:
Preserve strategic continuity.
Reduce repeated explanation.
Improve personalization.
Avoid repeated mistakes.
Retrieve relevant history.
Understand recurring patterns.
Recognize decision context.
Prepare better missions.
Improve prioritization.
Support long-term learning.
while preventing:
Stale-context errors.
Hidden authority.
Cross-project leakage.
Privacy exposure.
False certainty.
Memory-driven bias.
Contradiction.

### 22.11 Memory Layers

Executive Intelligence may use several conceptual memory layers:
1. Founder Memory.
2. Portfolio Memory.
3. Project Memory.
4. Mission Memory.
5. Workflow Memory.
6. Decision-Context Memory.
7. Incident Memory.
8. Organizational Learning Memory.
9. Temporary Working Memory.
Each layer should have different scope and retention.

### 22.12 Founder Memory

Founder Memory may preserve stable information such as:
Strategic preferences.
Communication style.
Risk tolerance.
Approval preferences.
Capacity patterns.
Long-term goals.
Repeated frustrations.
Preferred decision formats.
Founder Memory should be handled with strong privacy controls.

### 22.13 Portfolio Memory

Portfolio Memory may preserve:
Project relationships.
Shared strategic priorities.
Cross-project bottlenecks.
Portfolio allocation patterns.
Shared platform lessons.
Historical mode changes.
Systemic risks.
Portfolio Memory should normally use summaries rather than raw project data.

### 22.14 Project Memory

Project Memory may preserve:
Project purpose.
Target audience.
Brand identity.
Customer sensitivities.
Technical history.
Workflow lessons.
Performance patterns.
Known constraints.
Roadmap history.
Project Memory should remain isolated from other projects by default.

### 22.15 Mission Memory

Mission Memory may preserve:
What the mission attempted.
Which assumptions were used.
What blocked execution.
Which methods worked.
Which interventions occurred.
What outcome was achieved.
What should be reused later.
Mission state itself belongs in the operational system.
Durable lessons may be promoted to Memory.

### 22.16 Workflow Memory

Workflow Memory may preserve:
Successful configurations.
Known failure modes.
Optimal volume.
Required checks.
Provider weaknesses.
Common escalation cases.
Recovery patterns.
Workflow Memory should remain linked to workflow version and project.

### 22.17 Decision-Context Memory

Decision-Context Memory may preserve useful background around a formal decision.
Examples:
Founder concern.
Rejected alternatives.
Unresolved uncertainty.
Historical debate.
Contextual preference.
The formal decision remains in the Decision Ledger.

### 22.18 Incident Memory

Incident Memory may preserve:
Root cause.
Damage pattern.
Early warning signal.
Containment method.
Recovery lesson.
Recurring vulnerability.
Human response lesson.
Incident authority and formal outcomes should remain linked to incident records and the Decision Ledger.

### 22.19 Organizational Learning Memory

Organizational Learning Memory may contain abstracted lessons that are useful beyond one event.
Examples:
Approval packages with explicit alternatives are reviewed faster.
Shared workflows need hard project-context resets.
High-volume publishing requires stronger duplicate protection.
Founder review burden is often underestimated.

### 22.20 Temporary Working Memory

Temporary Working Memory supports the current reasoning process.
It may contain:
Current conversation context.
Open questions.
Intermediate comparisons.
Unconfirmed hypotheses.
Draft recommendations.
Temporary Working Memory should not automatically become durable memory.

### 22.21 Memory Classes

A memory may be classified as:
Fact.
Preference.
Constraint.
Pattern.
Lesson.
Hypothesis.
Risk.
Relationship.
Context.
Procedure.
Warning.
Personal boundary.
The class affects how the memory should be used.

### 22.22 Fact Memory

A Fact Memory represents a claim believed to be true.
Examples:
The Prompt is deployed on Vercel.
Familje-Stunden uses a subscription model.
GainPilot is currently hibernated.
Facts should include source and freshness.

### 22.23 Preference Memory

A Preference Memory represents a stable founder or project preference.
Examples:
The founder prefers Swedish discussion and English canonical documents.
The founder prefers explicit Git branch safety.
The founder wants material approvals to be decision-ready.
Preferences should not override policy or objective evidence.

### 22.24 Constraint Memory

A Constraint Memory represents a recurring limitation.
Examples:
Founder time is limited during a study period.
A project must remain hibernated.
Production changes require approval.
A workflow cannot safely retry external actions.
Current validity should be checked.

### 22.25 Pattern Memory

A Pattern Memory represents recurring behavior or correlation.
Examples:
Architecture missions often expand beyond original scope.
Low-capacity days produce slower approval decisions.
The Prompt performs better with current-event content.
Patterns should include evidence strength and confidence.

### 22.26 Lesson Memory

A Lesson Memory preserves a conclusion from experience.
Example:
Do not retry a publishing request when the external state is unknown.
Lessons should remain linked to their evidence.

### 22.27 Hypothesis Memory

A Hypothesis Memory preserves an unconfirmed belief.
Example:
Publishing more short news may improve newsletter growth.
Hypotheses must never be presented as established truth.

### 22.28 Risk Memory

A Risk Memory preserves a known recurring exposure.
Examples:
Cross-project context contamination.
Founder approval overload.
Unclear workflow ownership.
Duplicate external actions.

### 22.29 Relationship Memory

Relationship Memory may preserve:
A project depends on a shared capability.
A workflow supports a particular mission.
A founder preference applies to a specific decision class.

### 22.30 Context Memory

Context Memory preserves information needed to understand other records.
Example:
The Prompt was selected first because it already had an operational content pipeline.

### 22.31 Procedure Memory

Procedure Memory preserves known safe ways to perform recurring work.
It should not replace active workflow definitions.

### 22.32 Warning Memory

A Warning Memory highlights something that should be surfaced when relevant.
Example:
Do not merge major Omnira execution-engine changes before Codex review.
Warnings should remain scoped and current.

### 22.33 Personal Boundary Memory

Personal Boundary Memory may preserve founder-defined limits such as:
Protected family time.
Preferred notification windows.
Maximum approval burden.
Sensitive personal information that must not be exposed.
These memories require strict privacy.

### 22.34 Memory Scope

Every durable memory should have explicit scope.
Possible scopes:
Founder Private.
Global Omnira.
Portfolio.
Project.
Mission.
Workflow.
Role.
Customer Tenant.
Temporary Session.

### 22.35 Founder-Private Scope

Founder-private memories may be used by Atlas and authorized Executive functions.
They should not be exposed directly to:
Project Executives.
Manager.
Workforce.
Agents.
External tools.
unless explicitly authorized.

### 22.36 Global Omnira Scope

Global memories should contain only information appropriate across the organization.
Examples:
Core architectural principles.
General approval preferences.
Shared governance lessons.
Organization-wide terminology.

### 22.37 Portfolio Scope

Portfolio memories may span projects without exposing raw project-sensitive details.

### 22.38 Project Scope

Project memories should remain unavailable to other projects unless explicitly abstracted or authorized.

### 22.39 Mission Scope

Mission memories should normally be available only to the mission, Manager, relevant Workforce, and supervising Executive.

### 22.40 Workflow Scope

Workflow memories should remain tied to:
Project.
Workflow identity.
Workflow version.
Action category.

### 22.41 Customer-Tenant Scope

Future customer memories must remain tenant-isolated.
Internal Omnira memories must not silently influence customer organizations unless permitted.

### 22.42 Memory Provenance

Every memory should identify where it came from.
Possible sources:
Founder statement.
Approved decision.
Mission outcome.
Workflow telemetry.
Performance analysis.
Incident review.
Imported document.
Human correction.
System inference.

### 22.43 Direct vs Inferred Memory

A Direct Memory comes from an explicit source.
An Inferred Memory is derived from patterns or interpretation.
The system should distinguish them.
Example:
Direct:
The founder says newsletters should require approval.
Inferred:
The founder generally prefers higher review for external communication.

### 22.44 Inference Confidence

Inferred memories should include confidence.
Possible levels:
High.
Medium.
Low.
Unconfirmed.

### 22.45 Memory Evidence

A memory may link to:
Conversation.
Decision.
Report.
Metric.
Incident.
Approval history.
Mission result.
Document.
The evidence should remain accessible where authorized.

### 22.46 Memory Freshness

Every memory should have a freshness state.
Possible states:
Current.
Likely Current.
Needs Review.
Stale.
Expired.
Contradicted.

### 22.47 Memory Validity Horizon

Different memory classes should have different validity horizons.
Examples:
Founder communication preference:
Long-lived.
Current project mode:
Short-lived.
Temporary deadline:
Very short-lived.
Architectural principle:
Long-lived but reviewable.
Market assumption:
Short-lived.

### 22.48 Memory Age Is Not Validity

An old memory may remain valid.
A new memory may already be wrong.
Validity should depend on:
Source.
Context.
Volatility.
Contradiction.
Review.
Current evidence.

### 22.49 Memory Review Date

Material memories should have review dates when:
They affect strategy.
They describe current project state.
They influence autonomy.
They represent sensitive founder preferences.
They encode recurring constraints.

### 22.50 Memory Expiration

Some memories should expire automatically.
Examples:
Temporary campaign deadline.
Short-term founder availability.
Trial-specific constraint.
Temporary provider outage.
One-time exception.

### 22.51 Memory Contradiction

A contradiction exists when two memories cannot both be true in the same scope and time.
Example:
Memory A:
GainPilot is active.
Memory B:
GainPilot is hibernated.
The system should not choose silently.

### 22.52 Contradiction Resolution

Resolution should consider:
Source authority.
Timestamp.
Scope.
Directness.
Decision Ledger.
Current project state.
Founder confirmation.

### 22.53 Current State Wins Over Historical Memory

Operational truth should come from current authoritative systems where available.
Examples:
Current project mode.
Current license status.
Current budget.
Current workflow version.
Current policy.
Memory provides context around that state.

### 22.54 Decision Ledger Precedence

When a memory conflicts with an active Decision Ledger entry:
The active decision governs.
The memory should be marked stale, historical, or contradicted.

### 22.55 Policy Precedence

When memory conflicts with active policy:
Policy governs.
Memory may explain historical preference but cannot authorize action.

### 22.56 Founder Correction

The founder should be able to correct a memory directly.
The correction should:
Update current interpretation.
Preserve history where useful.
Record source.
Update confidence.
Propagate to dependent recommendations.

### 22.57 Memory Retraction

A memory may be retracted when:
It was incorrect.
It was inferred wrongly.
Consent is withdrawn.
It is no longer useful.
It creates privacy risk.
The user requests deletion.

### 22.58 Memory Versioning

Material memory changes should preserve:
Previous value.
New value.
Reason.
Source.
Time.
Affected scope.

### 22.59 Memory Consolidation

Repeated related events may be consolidated into a durable memory.
Example:
Several approvals show the founder consistently removes speculative claims.
Consolidated memory:
The founder prefers claims to remain conservative unless evidence is strong.
Consolidation should preserve underlying evidence.

### 22.60 Consolidation Threshold

Consolidation should require:
Repeated evidence.
Stable pattern.
Relevant future value.
Low contradiction.
Appropriate privacy.

### 22.61 Over-Consolidation

Over-consolidation may create overly broad beliefs.
Example:
Several edits in one project
→ Incorrect conclusion that the founder dislikes all long-form content.

### 22.62 Memory Summarization

Memory should compress detail while preserving meaning.
A good summary should retain:
Scope.
Context.
Source.
Confidence.
Exceptions.
Review needs.

### 22.63 Lossy Summarization Risk

Summarization may remove:
Conditions.
Exceptions.
Uncertainty.
Temporal limits.
Project scope.
Material memories should not be compressed beyond safe interpretation.

### 22.64 Memory Retrieval

Executive Intelligence should retrieve memories based on relevance, not simply recency.
Retrieval may consider:

### 22.65 Retrieval Query Context

A retrieval request should contain:

### 22.66 Minimum Necessary Retrieval

### 22.67 Retrieval Ranking

Memory ranking may consider:

Freshness.
Authority.
Confidence.

### 22.68 Salience

Salience describes how important a memory is likely to be.

Cross-project leverage.
Incident severity.

### 22.69 Salience Is Not Authority

A highly salient memory may deserve retrieval.

### 22.70 Retrieval by Project

Project-level reasoning should first retrieve:

Relevant global principles.
Current decisions.
Relevant founder preferences.

### 22.71 Retrieval by Decision Class

A pricing decision may retrieve:

### 22.72 Retrieval by Mission

### 22.74 Retrieval by Founder Preference

Source.
Freshness.

### 22.83 Durable Preference Update

Project preference.

### 22.84 Memory Promotion

### 22.86 Human-Confirmed Memory Promotion

### 22.88 Memory Deletion

### 22.98 Executive Access

### 22.99 Portfolio Access

Portfolio Executive should receive:

Shared lessons.

### 22.100 Project Executive Access

Failure Mode 10 — Preference Lock-In
