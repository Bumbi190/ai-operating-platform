## Canonical Text

### 26.1 Purpose of This Chapter

Executive Intelligence must understand how the organization fits together.
It must be able to see:
Which projects exist.
Which missions are active.
Which workflows execute them.
Which agents and roles participate.
Which tools and data are used.
Which decisions authorized the work.
Which evidence supported those decisions.
Which outcomes resulted.
Which risks, dependencies, and bottlenecks exist.
Which lessons should influence the next recommendation.
Traditional lists and dashboards are insufficient for many of these questions.
A list may show every workflow.
It may not show which mission each workflow serves.
A dashboard may show revenue.
It may not show which decisions, campaigns, workflows, and dependencies contributed to it.
Omnira therefore requires two complementary graph systems:
The Operating Graph.
The Intelligence Graph.
Together, they create organizational transparency without collapsing execution, reasoning, authority, and evidence into one ambiguous structure.

### 26.2 The Core Thesis

The Operating Graph explains:
How the organization is currently arranged and executing.
The Intelligence Graph explains:
Why the organization is acting, what evidence supported it, and what happened afterward.
The foundational principle is:
Operating Graph:
Who and what is doing the work?
Intelligence Graph:
Why is the work being done, what supports it, and what did it produce?

### 26.3 Two Graphs, Not One

Omnira should not force every organizational relationship into one undifferentiated graph.
Execution relationships and intelligence relationships have different meanings.
Example:
Project
→ Mission
→ Workflow
→ Agent
→ Tool
is operational.
While:
Evidence
→ Recommendation
→ Decision
→ Mission
→ Outcome
→ Review
→ Learning
is intellectual and institutional.
The two graphs should connect.
They should not lose their distinct purposes.

### 26.4 The Operating Graph

The Operating Graph represents how work is structured and executed.
Its primary chain is:
Project
→ Mission
→ Workstream
→ Workflow
→ Task
→ Role
→ Agent
→ Tool
→ Action
It may also represent:

Schedules.
Queues.
Budgets.

Approval gates.

### 26.5 The Intelligence Graph

The Intelligence Graph represents how reasoning, decisions, and learning evolve.
Its primary chain is:
Evidence
→ Interpretation
→ Recommendation
→ Approval
→ Decision
→ Mission
→ Action
→ Outcome
→ Review
→ Learning
→ Next Recommendation
It may also represent:
Assumptions.
Risks.
Alternatives.
Policies.
Memories.
Knowledge.
Trust Scores.
Autonomy Licenses.
Performance Metrics.
Opportunity Costs.

### 26.6 Graphs as Organizational Maps

The graphs should function as maps.
A map helps answer:
Where am I?
What is connected?
What depends on this?
What happens if this changes?
Who owns this?
Why does this exist?
What should I inspect next?
A graph should not merely be a visual decoration.

### 26.7 Graphs as Machine-Readable Context

The same graph relationships should support machine reasoning.
Executive Intelligence may ask:
Which active missions depend on this provider?
Which workflows use this credential?
Which decisions relied on this stale assumption?
Which outcomes came from this autonomy license?
Which project would be affected if this service failed?

### 26.8 Graphs as Explainability Infrastructure

A graph can explain a recommendation by showing its path.
Example:
Traffic decline
→ Search visibility decline
→ Outdated article cluster
→ Content freshness mission
→ Recommended workflow adjustment.
The graph makes the reasoning inspectable.

### 26.9 Graphs Are Not Authority

A graph relationship does not authorize action.
Example:
Workflow
→ Connected To
→ Publishing Tool
does not mean:
Workflow may publish autonomously.
Authority still comes from:
Policy.
Approval.
Mission authority.
Autonomy license.
Decision Ledger.
Budget mandate.

### 26.10 Graphs Are Not Truth by Themselves

A graph edge may be:
Verified.
Inferred.
Historical.
Proposed.
Stale.
Contradicted.
The graph must preserve evidence and confidence.

### 26.11 Graphs Are Not the Database

The graph may reference operational records.
It should not become the only storage system for every object.
Source systems remain responsible for their authoritative state.

### 26.12 Graphs Are Not the User Interface Alone

The graph architecture includes:
Node definitions.
Relationship definitions.
Provenance.
Permissions.
Temporal state.
Confidence.
Versioning.
Queries.
Views.
Visualizations.
The visual graph is only one presentation layer.

### 26.300 Design Requirements

Any implementation of Executive Graphs & Transparency should preserve these requirements:
Omnira must maintain a conceptual separation between the Operating Graph and the Intelligence Graph.
The Operating Graph must represent how work is structured and executed.
The Intelligence Graph must represent evidence, reasoning, decisions, outcomes, and learning.
Every material node must have a stable identity.
Every material node and edge must have explicit scope.
Every material relationship must have provenance.
Every inferred relationship must communicate confidence.
The graph must distinguish current, historical, proposed, and target state.
Planned architecture must not be presented as deployed reality.
Capability and connectivity must not be confused with authority.
Authority relationships must reference valid decisions, policies, approvals, or licenses.
Project and tenant isolation must apply to graph storage, traversal, queries, views, and exports.
Founder-private and customer-sensitive information must use minimum necessary representation.
Executive, Manager, Workforce, and agents must receive role-appropriate subgraphs.
The graph must support backward and forward lineage.
Decisions must be traceable to evidence, authority, missions, outcomes, reviews, and learning.
Actions must be traceable to project, mission, workflow, role, tool, and authority.
Graph views must support progressive disclosure.
Visual meaning must remain explicit and accessible.
Graph data must remain versioned and temporal.
Orphan, shadow, stale, contradictory, and unowned objects must be detectable.
The graph must degrade safely when unavailable.
Source systems must remain authoritative for their own current state.
Graph-based recommendations must remain explainable.
The graph must improve real governance and understanding rather than serve as visual decoration.
