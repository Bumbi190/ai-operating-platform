## Canonical Text

### 27.1 Purpose of This Chapter

Omnira requires a clear place where authorized humans can review and decide consequential actions.
That place is the Approval Inbox.
The Approval Inbox should allow an authorized user to:
Approve.
Reject.
Edit.
Approve with conditions.
Request evidence.
Request an alternative.
Request deeper analysis.
Defer.
Delegate review.
Escalate.
Withdraw a prior approval where still reversible.
It must provide enough context to answer:
What am I being asked to decide?
Why is this decision required?
Which project does it affect?
What will happen if I approve?
What will happen if I reject?
What evidence supports the recommendation?
What risks exist?
What authority am I granting?
How long will that authority remain valid?
Can the action be reversed?
What is the opportunity cost of waiting?
Is there a safer alternative?
The purpose of the Approval Inbox is not to create more approval work.
Its purpose is to ensure that approval work is necessary, clear, bounded, efficient, and authoritative.

### 27.2 The Core Thesis

An approval request should arrive as a decision-ready package.
The authorized user should not need to reconstruct:
Project context.
Mission context.
Technical history.
Relevant evidence.
Policy requirements.
Damage exposure.
Alternatives.
Consequences.
The foundational principle is:
The Approval Inbox should ask for judgment.
It should not ask the founder to perform investigation, coordination, or context reconstruction.

### 27.3 Approval as an Authority Act

An approval is not merely positive feedback.
An approval is an authority act.
It may authorize:
A specific external action.
A specific internal change.
A bounded spend.
A production deployment.
A customer communication.
A publishing action.
A project-mode transition.
An autonomy license.
A strategic decision.
Because approval creates authority, it must be:
Scoped.
Explicit.
Traceable.
Time-aware.
Object-specific.
Condition-aware.
Revocable where possible.

### 27.4 Approval Is Not Agreement

A user may agree with an idea without authorizing execution.
Example:
Statement:
This newsletter concept is good.
Not necessarily:
Send the newsletter.
The interface should distinguish:
Agree with direction.
Approve preparation.
Approve implementation.
Approve external execution.
Approve recurring autonomy.

### 27.5 Approval Is Not Review Completion

A reviewer may finish reviewing an item and still choose:
Reject.
Request evidence.
Defer.
Escalate.
Request an alternative.
Review completion and approval outcome are separate.

### 27.6 Approval Is Not Editing

Editing an artifact changes its content.
Approval authorizes the resulting object or action.
The system should not assume that editing automatically approves.
After an edit, the user should be able to choose:
Save edit only.
Save and approve.
Save and request revalidation.
Save and return to preparation.

### 27.7 Approval Is Not Policy

One approval authorizes a bounded object or action.
It does not automatically create a reusable policy.
Example:
Approved:
Send this newsletter.
Not automatically approved:
Send all future newsletters.
Repeated approval may support a future policy or autonomy recommendation.
It does not create one silently.

### 27.8 Approval Is Not Autonomy

Approval may authorize one action.
Autonomy authorizes a category of actions under a license.
The progression is:
Individual Action Approval
→
Batch Approval
→
Policy-Bound Approval
→
Autonomy License.
Each progression requires separate authority.

### 27.9 Approval Is Not Legal Permission

Human approval cannot make a prohibited action lawful, safe, or policy-compliant.
An approval should not override:
Constitutional restrictions.
Legal prohibitions.
Tenant isolation.
Security boundaries.
Unavailable authority.
Non-delegable requirements.
Where approval conflicts with higher rules, Governance must deny or escalate.

### 27.10 Approval Is Not Silence

No response is not approval.
No response may mean:
The user has not reviewed the item.
The user is unavailable.
The request lacks urgency.
The request is unclear.
The decision should expire or remain pending.
Silence must never create external authority.

### 27.11 The Approval Inbox Objective

The Approval Inbox should:
Reduce founder coordination burden.
Concentrate consequential decisions.
Present decision-ready context.
Prevent hidden authority.
Support fast low-risk approvals.
Support deep high-impact review.
Preserve project isolation.
Connect decisions to execution.
Create evidence for future autonomy.
Keep routine execution quiet.

### 27.20 Requested Authority

Send one newsletter.
Deploy one reviewed release.

Activate one mission.
Approve one migration.

### 27.22 Version-Pinned Approval

Approval should apply to the reviewed version.
If the object changes materially afterward:
Approval becomes invalid.
Revalidation is required.
A new approval version is created.

### 27.25 Executive Summary

### 27.28 Evidence Summary

Limitations.

### 27.30 Alternatives

Approve as proposed.
Approve narrower scope.
Approve a trial.
Request revision.
Do nothing.

### 27.31 Status Quo Alternative

The request should explain the consequence of not acting.

### 27.32 Risk

Likelihood.
Impact.
Reversibility.

Detection.
Recovery.

### 27.33 Damage Boundary Assessment

Remains below the Damage Boundary.

### 27.34 Opportunity Cost

The request should state what approval or rejection may displace.
Examples:
Founder time.
Project capacity.
Budget.
Roadmap sequence.
Another campaign.
A safer alternative.

### 27.35 Cost

Cost may include:
Direct spend.
AI cost.
Infrastructure.
Founder review.
Workforce effort.
Maintenance.

### 27.36 Deadline

The request should state whether the deadline is:
Hard.
Soft.
Market-driven.
Operational.

Self-imposed.

### 27.37 Reversibility

The request should state whether the action is:
Fully reversible.
Partially reversible.
Operationally difficult to reverse.
Externally irreversible.

### 27.38 Rollback

Where rollback is possible, the request should define:
How to reverse.
Who may reverse.
How quickly.
What remains after reversal.
How success is verified.

### 27.39 Conditions

Approval may include conditions.
Examples:
Limit to one channel.
Limit to ten recipients.
Use the reviewed version only.
Spend no more than the approved amount.
Require monitoring.

Exclude sensitive categories.

### 27.40 Expiration

Approval should expire when:
The object changes.
The deadline passes.
The project mode changes.
The policy changes.
The workflow version changes.
The evidence becomes stale.
The approval window ends.

### 27.44 Draft

A Draft request is incomplete and should not reach the approver.

### 27.45 Preparing

The system is assembling:
Evidence.
Alternatives.
Risk.
Context.
Required validation.

### 27.46 Pending

A Pending request is ready for review.

### 27.47 Needs Evidence

The approver has requested more evidence.
No authority has been granted.

### 27.48 Needs Revision

The approved object or proposal requires modification.

### 27.49 Deferred

The decision is intentionally postponed.
A defer action should include:
Reason.
Review date.
Trigger condition.
What may continue.
What remains blocked.

### 27.50 Escalated

The request has been transferred to a higher or specialist authority.

### 27.51 Approved

The exact requested authority has been granted.

### 27.52 Approved with Conditions

Authority has been granted subject to explicit conditions.
Conditions should be machine-readable where possible.

### 27.53 Rejected

The requested authority has been denied.
Rejection should preserve:
Reason.
Whether revision is allowed.
Whether an alternative is preferred.
Whether the request may return later.

### 27.54 Withdrawn

The requester no longer seeks approval.

### 27.55 Expired

The approval or request is no longer valid because time, object, state, or context changed.

### 27.56 Executed

The approved action has been performed and verified.

### 27.57 Failed

Execution failed after approval.
Failure should not imply that approval was necessarily incorrect.

### 27.58 Superseded

A newer Approval Object has replaced the current one.

### 27.59 Cancelled

The underlying mission, decision, or action no longer exists.
Part III — Core Approval Actions

### 27.60 Approve

Approve should grant exactly the authority described.
The interface should confirm:
Object.
Scope.
Conditions.
Expiration.
Downstream action.

### 27.61 Approve Immediately

Immediate approval may be appropriate for:
Clear low-risk requests.
Time-sensitive but reversible actions.
Well-understood recurring decisions.
Decision-ready packages with strong evidence.

### 27.62 Approve with Conditions

This action should allow the approver to narrow or control execution.
Possible conditions:
Smaller scope.
Lower spend.
Shorter duration.
Limited audience.
Extra monitoring.
Additional review.
No automatic renewal.
Specific rollback threshold.

### 27.63 Conditional Approval Enforcement

Conditions must be enforced technically.
They should not exist only as free-text comments.

### 27.64 Reject

Reject should block the requested authority.
The system should ask whether the rejection is:
Final.
Final for current version.
Temporary.
Dependent on missing evidence.
A preference for another alternative.

### 27.65 Reject with Reason

A reason improves:
Learning.
Future request quality.
Trust calibration.
Policy design.
Executive understanding.

### 27.66 Edit

Edit allows the approver to modify the reviewed object.
Examples:
Article text.
Newsletter subject.
Mission scope.
Budget amount.
License conditions.
Customer reply.
Policy wording.

### 27.67 Editing Materiality

Edits should be classified as:
Cosmetic.
Minor.
Material.
Authority-changing.
Risk-changing.

### 27.68 Cosmetic Edit

A cosmetic edit changes presentation without changing meaning.
It may preserve existing validation.

### 27.69 Minor Edit

A minor edit changes wording but not material claims, scope, or risk.

### 27.70 Material Edit

A material edit changes:
Facts.
Recommendation.
Scope.
Audience.
Spend.
Promise.
Authority.
Risk.
Material edits should trigger revalidation.

### 27.71 Authority-Changing Edit

An edit that broadens authority must create a new approval request.
The approver should not silently grant authority beyond what the system evaluated.

### 27.72 Risk-Changing Edit

An edit that increases risk should trigger:
New Damage Boundary assessment.
New policy evaluation.
Possible specialist review.

### 27.73 Save without Approval

The user should be able to edit and save without authorizing action.

### 27.74 Edit and Approve

This action should approve the exact edited version.

### 27.75 Edit and Return for Validation

This action should allow the system to:
Recheck facts.
Recheck policy.
Recheck scope.
Recalculate risk.
Update the summary.

### 27.76 Request Evidence

The approver may request:
More sources.
Test results.
Performance data.
Customer evidence.
Cost analysis.
Legal reference.
Technical validation.
Rollback proof.

### 27.77 Evidence Request Scope

The evidence request should state:
What is missing.
Why it matters.
What quality is required.
Whether work may continue meanwhile.

### 27.78 Request Alternative

The approver may ask for:
A safer alternative.
A cheaper alternative.
A faster alternative.
A narrower alternative.
A non-AI alternative.
A reversible trial.
A no-action option.

### 27.79 Compare Alternatives

The interface should allow comparison across:
Expected value.
Cost.
Risk.
Reversibility.
Time.
Founder burden.
Strategic fit.
Opportunity cost.

### 27.80 Request Deep Analysis

The user may escalate the reasoning depth.
Possible request:
Reevaluate using Deep reasoning.
Run a pre-mortem.
Compare three scenarios.
Verify the evidence independently.
Prepare a Board-level decision package.

### 27.81 Deep Analysis Is Not Automatic Delay

Deep analysis should be used when additional reasoning may materially improve the decision.

### 27.82 Defer

Defer should postpone the decision without rejecting it.
A defer action should define:
Until date.
Until condition.
Until evidence.
Until capacity exists.
Until another decision is resolved.

### 27.83 Defer with Safe Continuation

Where possible, safe internal work may continue.
Examples:
Drafting.
Testing.
Research.
Preview deployment.
Alternative preparation.

### 27.84 Escalate

The user may escalate to:
Portfolio Executive.
Founder.
Security reviewer.
Legal reviewer.
Financial reviewer.
Project owner.
Crisis authority.

### 27.85 Delegate Review

A future multi-user organization may allow review delegation.
Delegation should specify:
Review authority.
Decision authority.
Scope.
Duration.
Restrictions.

### 27.86 Withdraw Approval

An approver may withdraw an approval if:
Execution has not occurred.
The action remains reversible.
New evidence appears.
Conditions were violated.
The context changed.

### 27.87 Withdrawal after Execution

After execution, withdrawal may become:
Rollback request.
Unpublish request.
Cancellation.
License suspension.
Incident response.
Part IV — Decision-Ready Approval Packages

### 27.88 Decision-Ready Standard

An approval request should be considered decision-ready when:
The decision is explicit.
The object is fixed.
The authority is clear.
Evidence is sufficient.
Risks are visible.
Alternatives are visible.
Consequences are understood.
The approver can act without investigation.

### 27.89 One Decision per Request

An Approval Object should normally ask one material decision.
Bundling unrelated decisions creates confusion.

### 27.90 Compound Approval

A compound approval may be valid when actions are inseparable.
Example:
Approve migration and required deployment sequence.
The package should still separate each consequence.

### 27.91 Hidden Decisions

The request should not hide additional decisions inside one button.
Example:
Approve website launch
must not silently include:
Change pricing.
Send customer email.
Delete old content.
Increase budget.

### 27.92 Recommendation First

The request should clearly state the recommended decision before detailed evidence.

### 27.93 Evidence on Demand

The interface should support progressive disclosure:
Executive summary.
Evidence summary.
Detailed evidence.
Raw source.
Audit trace.

### 27.94 Risk on Demand

The user should be able to inspect:
Primary risk.
Worst credible outcome.
Residual risk.
Controls.
Rollback.
Specialist review.

### 27.95 Technical Detail on Demand

Technical information should be available without dominating the default view.

### 27.96 Change Summary

For changed artifacts, the request should show:
What changed.
Why.
Who changed it.
What risk changed.
Whether prior validation remains valid.

### 27.97 Diff View

Approvals involving code, documents, policies, or content should support a precise diff.

### 27.98 Before-and-After View

A before-and-after view may show:
Current state.
Proposed state.
Consequences.
Rollback.

### 27.99 Source View

The approver should be able to inspect source material when factual accuracy matters.

### 27.100 Confidence

The request should communicate confidence in:
Evidence.
Recommendation.
Forecast.
Attribution.
Risk estimate.

### 27.101 What Would Change the Recommendation

Material requests should explain:
Which new evidence would reverse the recommendation?
Which assumption is most fragile?
Which threshold would require a different decision?

### 27.102 Decision Consequence Map

The request may show a graph path:
Approval
→ Mission release
→ Workflow action
→ External effect
→ Monitoring
→ Review.

### 27.103 Rejection Consequence Map

The request should also show:
Rejection
→ Work stopped
→ Value delayed
→ Risk avoided
→ Alternative available.

### 27.104 Deferral Consequence Map

Deferral may result in:

Cost avoided.
Evidence improved.
Mission blocked.
Capacity preserved.

### 27.342 Failure Mode 3 — Approval without Object Version

### 27.343 Failure Mode 4 — Hidden Authority

The request does not state what approval grants.

### 27.344 Failure Mode 5 — Hidden Decisions

Several consequential choices are bundled inside one button.

### 27.345 Failure Mode 6 — Context Reconstruction

The founder must investigate before deciding.

### 27.346 Failure Mode 7 — Rubber Stamping

Excessive routine requests reduce real review quality.

### 27.347 Failure Mode 8 — Approval Theatre

The action occurred before approval or approval cannot change the outcome.

### 27.348 Failure Mode 9 — Unenforced Conditions

Conditions exist only as comments.

### 27.349 Failure Mode 10 — Permanent Approval

Authority remains valid after context, policy, or object changes.

### 27.350 Failure Mode 11 — Self-Approval

A workflow or Executive grants itself broader authority.

### 27.351 Failure Mode 12 — Cross-Project Leakage

The approval package exposes or authorizes another project improperly.

### 27.352 Failure Mode 13 — Editing Equals Approval

### 27.353 Failure Mode 14 — Approval Equals Policy

Part XXII — Design Requirements

### 27.356 Design Requirements

Any implementation of the Approval Inbox should preserve these requirements:
Every approval request must identify one clear decision.
Every request must identify the affected project.
Every request must identify the exact object or action being approved.
Every approval must be tied to an object version where material.
Every request must state the authority approval grants.
Every request must state why approval is required.
Every material request must include evidence, alternatives, risk, cost, reversibility, and opportunity cost.
Every request must communicate the consequence of approval, rejection, and deferral.
Every approval condition must be machine-enforceable where possible.
No response must never create authority.
Editing must remain separate from approval.
Material edits must trigger revalidation.
Approval must remain separate from policy and autonomy.
Repeated approvals may trigger a policy or autonomy recommendation but not automatic promotion.
Every approval must have explicit scope, conditions, and expiration where relevant.
Every approval must verify approver authority.
No workflow, agent, Manager, or Executive may approve its own higher authority.
Project and tenant isolation must apply to requests, evidence, decisions, and execution.
Sensitive founder and customer information must be minimized.
Every consequential action must verify a valid approval receipt before execution.
Every material approval must remain traceable to downstream execution and outcome.
Routine approval burden must be measured and reduced.
Approval interfaces must avoid coercive urgency or manipulative design.
High-impact approvals must support strong authentication and separation of duties.
The Inbox must degrade safely when unavailable.
The Approval Inbox must concentrate real authority without turning the founder into a permanent operational bottleneck.

Atlas conversational approval.
The technical implementation may evolve.
The approval contract must remain stable.

### 27.358 The Approval Inbox Contract

Omnira must:
Present consequential decisions in one clear, decision-ready surface.
Separate review, editing, approval, policy, and autonomy.
State exactly what authority each approval grants.
Bind approval to the reviewed object, version, project, scope, and conditions.
Provide evidence, alternatives, risk, cost, reversibility, and opportunity cost.
Allow approve, reject, edit, request evidence, request alternatives, deep analysis, and defer.
Prevent silence, ambiguity, or casual language from becoming authority.
Prevent self-approval and hidden authority expansion.
Enforce approval conditions technically.
Expire approvals when objects, policies, evidence, or context change.
Protect project, tenant, founder, customer, financial, and security boundaries.
Route decisions to the correct authority.
Support stronger authentication and multiple approvers where risk requires it.
Connect approval to execution, outcome, review, and learning.
Identify recurring approvals that should become batches, policies, or autonomy candidates.
Reduce founder burden without reducing human control.
Make the founder’s authority precise, understandable, and operationally real.
A feature that does not support this contract should not determine how Omnira requests or records human authority.

### 27.359 Closing Principle

The Approval Inbox should allow the founder to say:
I understand what you are asking.
I understand why you are asking me.
I understand what will happen if I approve.
I understand what will not be authorized.
I understand the evidence.
I understand the risk.
I understand the alternative.
I can decide without reconstructing the entire project.
It should also allow Omnira to say:
This action requires your judgment.
This action does not require your judgment.
This recurring decision is ready for batching.
This workflow may be ready for a bounded autonomy license.
This approval has expired.
This object changed and must be reviewed again.
This request is blocked because the required authority does not exist.
A mature Approval Inbox should not maximize approvals.
It should maximize legitimate, informed authority while minimizing unnecessary founder effort.
Individual approvals create precise control.
Batch approvals reduce repetition.
Policy-bound execution creates consistency.
Autonomy licenses create scalable independence.
The Approval Inbox is the bridge between them.
It is where Omnira proves that autonomy and human control are not opposites.
Human control becomes stronger when authority is explicit, evidence is clear, execution is traceable, and routine decisions no longer consume the founder’s attention.
