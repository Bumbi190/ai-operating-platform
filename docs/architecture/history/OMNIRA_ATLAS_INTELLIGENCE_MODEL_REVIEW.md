## 1. Final `IntelligenceObject` schema

The object is the single reusable unit every producer emits and every consumer reads. It is defined in TypeScript (`lib/atlas/intelligence/types.ts`) — that contract is the public architecture; the table is one implementation of it.

```ts
interface IntelligenceObject<B = Record<string, unknown>> {
  id:          string                 // assigned by the store
  kind:        IntelligenceKind       // 'brief' | 'trend' | 'entity_momentum' | 'reasoning' | 'executive_brief'
  subject:     Subject                // { kind: 'entity'|'content'|'project'|'global'; ref: string | null }
  projectId:   string | null          // project scope; null = platform-global
  summary:     string                 // one-line headline
  findings:    Finding[]              // structured conclusions, each with its own evidence
  body:        B                      // producer-specific structured payload (e.g. BriefBody)
  confidence:  number                 // 0–1, platform-wide convention (matches atlas.memories)
  evidence:    EvidenceChain          // complete chain: every input that informed the object
  producedBy:  string                 // producer identity, e.g. 'brief-producer'
  version:     string                 // producer semver, e.g. 'brief-producer-1.0.0'
  producedAt:  string                 // ISO; assigned by the store
  validUntil?: string | null          // optional staleness horizon
  supersededBy?: string | null        // lifecycle pointer; null = current
}

interface Finding  { label: string; detail: string; confidence?: number; evidence?: EvidenceChain }
type EvidenceChain = Evidence[]
interface Evidence { sourceKind: 'signal'|'memory'|'content'|'collector_run'|'intelligence'|'url'
                     refId: string; weight: number; observedAt: string; note?: string }
```

**Postgres mapping (`atlas_intelligence`)** — purely a storage detail behind `IntelligenceStore`:

| Domain field | Column | Notes |
|---|---|---|
| id | `id uuid` | pk |
| kind | `kind text` | |
| subject.kind / subject.ref | `subject_kind text` / `subject_ref text` | `subject_ref` is free-text, **no FK** |
| projectId | `project_id uuid` | FK → projects, nullable |
| summary | `summary text` | |
| findings | `findings jsonb` | |
| body | `body jsonb` | |
| confidence | `confidence numeric(4,3)` | 0–1 CHECK |
| evidence | `evidence jsonb` | the chain |
| producedBy / version | `produced_by text` / `version text` | |
| producedAt | `produced_at timestamptz` | |
| validUntil | `valid_until timestamptz` | |
| supersededBy | `superseded_by uuid` | self-ref; append-only lifecycle |

**Confidence** here is `0.88` = `0.1 base + 0.2×3 kinds + min(0.2, 0.02×4 signals=0.08) + 0.1 memory` — deterministic and explainable, never LLM-assigned. **Evidence chain** is complete: one entry per signal plus the memory item; the top-level chain is the union, and each finding carries its own slice. Every `refId` resolves to a real `atlas_signals` row or memory item, so any conclusion is fully auditable.

> Today the orchestrator's memory recall is a no-op seam, so live briefs run with `memoryUsed: 0` until a Memory Retrieval API exists. The shape above is exactly what appears once memory is injected — no producer change needed.

## 3. How `atlas_entities` is referenced

`atlas_entities` is the canonical registry; identity is `(kind, key)`. Nothing stores duplicate entity representations — they store **references** to it:

- **Subject:** when an object is about an entity, `subject = { kind: 'entity', ref: '<entity key>' }`. `subject_ref` holds the key; `getEntity({ kind, key })` resolves the canonical row.
- **Evidence / future relationships:** evidence `refId` and relationship endpoints use the same `(kind, key)` identifiers.
- **The example Brief** is project-scoped (`subject.kind = 'project'`), so it references a project, not an entity. An **Entity Intelligence** object would instead carry `subject = { kind: 'entity', ref: 'openai' }` pointing at `atlas_entities('company','openai')`.

Producers never invent entity rows ad hoc — they resolve/upsert through the store (`upsertEntity` / `getEntity`), keeping one representation per real-world thing. When a graph backend arrives, entities + relationships become nodes/edges behind the same references, with no change to objects that point at them.

## 4. How `getIntelligence()` returns a Brief

The read path is one door, fully decoupled from storage:

```
consumer
  → retrieval.getIntelligence(id)            // lib/atlas/intelligence/retrieval.ts
     → IntelligenceStore.getById(id)         // interface (store.ts)
        → PostgresIntelligenceStore.getById  // v1 impl: SELECT … WHERE id = $1
           → rowToObject()                   // jsonb columns → typed IntelligenceObject
  ← IntelligenceObject<BriefBody> | null
```

A consumer types the body to read brief-specific fields:

```ts
import { getIntelligence } from '@/lib/atlas/intelligence/retrieval'
import type { BriefBody } from '@/lib/atlas/intelligence/producers/brief-producer'

const brief = await getIntelligence<BriefBody>(id)
brief?.summary               // "4 signal(s) across 3 kind(s) · MRR 12450 kr · …"
brief?.confidence            // 0.88
brief?.body.metrics.mrrSek   // 12450
```

`getEvidenceChain(id)` returns just the chain for an audit view; `queryIntelligence(query)` is the bulk Context Retrieval (latest-only by default). None of these mention Postgres — they call the interface.

## 5. How Manager consumes it

Manager (and any agent) treats intelligence as **read-only, pre-computed** objects — it never recomputes or reaches into `atlas_signals` for synthesis:

```ts
// "What's the latest on GainPilot?"
const [brief] = await queryIntelligence<BriefBody>({
  subject: { kind: 'project', ref: 'proj-gainpilot' },
  kinds: ['brief'],
  limit: 1,
})

// Manager renders / reasons over the refined object:
brief.summary                  // headline for the user
brief.findings                 // structured points (revenue, audience, content_impact, …)
brief.confidence               // how much to trust it (gate low-confidence in UI)
// drill-down / "why?": resolve the chain
const chain = await getEvidenceChain(brief.id)   // → signals + memory behind it
```

Manager depends only on `retrieval.ts` + the contracts. It can filter by `minConfidence`, scope by project/entity, and trace any claim to its evidence. Same objects serve future agents and The Prompt — one intelligence layer, many consumers.

## 6. How future producers fit the same model

Every producer is a pure function returning an `IntelligenceDraft`, persisted via `IntelligenceStore.record`. They differ only in `kind`, `subject`, `body`, and how they build findings/confidence/evidence — the object shape, store, and read API are identical.

| Producer | kind | subject | body (B) | reads (evidence sources) |
|---|---|---|---|---|
| **Brief** (P1, done) | `brief` | project / global | `BriefBody` (counts, metrics, window) | signals (+ memory) |
| **Trend** | `trend` | entity / project / global | direction, slope, inflection, window | `getSignalTimeSeries` → signal points |
| **Entity Intelligence** | `entity_momentum` | entity (`atlas_entities` ref) | momentum, standing, related entities | signals + memory + (later) relationships |
| **Executive Brief** | `executive_brief` | global / project | the 5 exec answers | other intelligence objects (`sourceKind: 'intelligence'`) + signals |

Two structural points this enables without rework: producers can build **on each other** (Executive Brief cites Brief/Trend objects via `sourceKind: 'intelligence'`, forming evidence chains across the layer), and adding a producer is additive — a new pure module + a new `kind`, no schema or read-API change. The exclusions you set (Trend, Relationship, Graph) stay out until their milestones; the model already has a clean slot for each.

## 7. Storage-agnostic confirmation

The design is storage-agnostic by construction:

```
producers (buildBrief, future Trend/Entity/Exec)        consumers (Manager, agents, The Prompt)
        │ emit IntelligenceDraft                                  │ call retrieval.ts
        └───────────────►  IntelligenceStore (interface)  ◄───────┘
                                     ▲
                 ┌───────────────────┴───────────────────┐
        PostgresIntelligenceStore (v1)            GraphifyStore (future)
        atlas_intelligence / atlas_entities       graph nodes / edges
```

- Producers depend on the **contracts** (`types.ts`) only — no DB import. `buildBrief` is pure and has zero storage knowledge.
- Consumers depend on **`retrieval.ts`**, which depends on the **`IntelligenceStore` interface** — not on Postgres.
- The only Postgres-aware file is `postgres-store.ts`. `createIntelligenceStore()` is the single swap point.

**To introduce Graphify**, implement `IntelligenceStore` (`record`, `supersede`, `getById`, `getLatest`, `query`, `upsertEntity`, `getEntity`) over the graph backend and switch `createIntelligenceStore()`. Entities become nodes, relationships become edges, evidence becomes provenance edges — and **no producer or consumer changes**. Graphify is evaluated as a future implementation of Atlas Memory / Intelligence, not part of any current milestone.
